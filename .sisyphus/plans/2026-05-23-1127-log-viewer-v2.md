## Context

### Original Request

User feedback after V1 log viewer shipped: "Logs are too verbose — hard to focus. No timestamps, no color coding. The container is tiny. Can we open logs in a completely new page?"

### Interview Summary

**Key Discussions**:

- **Raw log analysis**: 148-line sample showed ~85% is OpenCode server internal noise. Only ~15% is useful (harness lifecycle, errors, LLM calls).
- **Layout**: Full-page route (not inline). Navigates from task detail page.
- **Filtering**: Client-side JSON parsing + filtering. Default shows important lines only, toggle for verbose.
- **Features**: Formatted columns, color coding, search/text filter, download raw, auto-scroll toggle, line count/stats.
- **Inline viewer**: Remove entirely — replace with link/button to full-page route.

**Research Findings**:

- Log lines are pino-structured JSON: `{"level":30,"time":"...","component":"...","msg":"..."}`
- 3 log levels: 30=INFO, 40=WARN, 50=ERROR (no DEBUG/TRACE)
- `component` field always present. Key components: `opencode-harness`, `opencode-server`, `session-manager`, `postgrest-client`
- LLM error lines can be ~65,000 chars (includes full prompt in error response)
- SSE endpoint never sends `event: done` for running tasks — streams indefinitely
- Dashboard uses React Router, all routes inside `<Layout>` (sidebar/nav stays)

### Metis Review

**Identified Gaps** (addressed):

- **65KB line truncation**: Must truncate display to ~500 chars with expandable "Show full"
- **Precise noise filter criteria defined**: SIGNAL vs NOISE components/services identified
- **Live stats**: Stats bar must update in real-time as lines arrive during streaming
- **Route nesting**: Inside `<Layout>` — sidebar stays (consistent with all other routes)

  **Implementation steps**:
  1. Define `ParsedLogEntry` interface:
     ```ts
     interface ParsedLogEntry {
       raw: string; // original line
       timestamp: string; // formatted as "HH:MM:SS.mmm"
       level: 'info' | 'warn' | 'error';
       component: string; // e.g. "harness", "server", "session-mgr"
       message: string; // the msg field
       isSignal: boolean; // true = important, false = noise
     }
     ```
  2. Export `parseLine(raw: string): ParsedLogEntry` function:
     - Try `JSON.parse(raw)`. If it fails, return a fallback entry with `raw` as message, level='info', component='unknown', isSignal=true.
     - Extract `time` → format as `HH:MM:SS.mmm` (parse ISO string, extract time part).
     - Extract `level` → map: 30='info', 40='warn', 50='error'. Default to 'info' for unknown.
     - Extract `component` → shorten: `'opencode-harness'` → `'harness'`, `'opencode-server'` → `'server'`, `'session-manager'` → `'session-mgr'`, `'postgrest-client'` → `'postgrest'`. Keep others as-is.
     - Extract `msg` → the message string.
     - Compute `isSignal`:
       - TRUE if: `component === 'opencode-harness'` OR `component === 'session-manager'` OR `component === 'postgrest-client'` OR `level >= 40` OR (`component === 'opencode-server'` AND `msg` matches `/service=(llm|session\.prompt|session\.processor|bash-tool)/`)
       - FALSE otherwise
  3. Export `parseLines(rawLines: string[]): ParsedLogEntry[]` — maps over array.
  4. Export `MAX_DISPLAY_CHARS = 500` constant.
  5. Export `truncateMessage(msg: string, max?: number): { text: string; truncated: boolean }` helper.
