## Context

### Original Request

The employee generated by the prior plan (`77e77c86`, cleaning-schedule-v16) works, but the user inspected it in the dashboard and found two architectural problems that violate the product vision (a non-technical user types plain English; the AI employee figures out the mechanics itself):

### Research Findings (verified by exploration)

- **Base worker AGENTS.md** `src/workers/config/agents.md` is thin (safety rails + Composio discovery). The compiler `agents-md-compiler.mts` injects it LAST as `## Platform Rules` (position 9 — lowest reading priority). High-priority slots are compiler-hardcoded sections (positions 2–4: `CRITICAL_DIRECTIVE`, the `<execution-instructions>` wrapper, the `<delivery-instructions>` wrapper).
- **Date is already auto-injected** (`prompt-assembler.mts` prepends `TODAY: ... | EPOCH_MS: ...`). **`<approved-content>` is already platform-owned** (`delivery-phase.mts` injects it unconditionally). **`submit-output` / `EXECUTION_PROMPT`** are already harness-injected. → Teaching these in `execution_steps` is REDUNDANT; removing the teaching is zero runtime risk.
- **Output contract single source** `src/lib/output-contract-constants.ts`: `DRAFT_PATH = '/tmp/draft.txt'`, `SUMMARY_PATH`, `APPROVAL_MESSAGE_PATH`. The generator-taught `/tmp/delivery-draft.txt` is a DIFFERENT path and NOT a platform constant. Delivery reads `deliverable.content` from the `deliverables` DB table — NOT any `/tmp` file. So `/tmp/*draft*.txt` is execution scratch only (to be validated in W1).
- **Skills are not auto-loaded**; OpenCode loads on demand. `slack` and `tool-usage-reference` skills already exist in the worker image; `tool-usage-reference` is never pruned. `filterCustomSkills()` prunes unconnected services.
- **Generator prompt** `src/gateway/services/prompts/archetype-generator-prompts.ts` already contains a CORRECT live-fetch "Notion Data Extraction Pattern" — but it is CONTRADICTED by a "Concrete Execution Steps Example" that says "Do NOT read from Notion, this table IS the source," a 10-point list (items 4 & 7 mandate hardcoded coverage/calendar tables), and "Explicit Business Rules Encoding." `buildConverseSystemPromptPre(isCreate:true)` is a near-verbatim duplicate (with an 11th item — drift). `REFINE_SYSTEM_PROMPT_PRE` teaches NONE of these patterns. The existing Notion pattern assumes COLUMNAR data — it must be made prose-aware.
- **Tools**: Notion read = `composio/execute.ts --action NOTION_GET_PAGE_MARKDOWN --params '{"page_id":...}'` → raw envelope, content at `.data.markdown` (needs `COMPOSIO_API_KEY` + tenant Notion OAuth; HTTP 400/exit 1 with no fallback if absent). Hostfully = `hostfully/get-checkouts.ts --date YYYY-MM-DD` (needs `HOSTFULLY_API_KEY` + `HOSTFULLY_AGENCY_UID` from `tenant_secrets`; VLRE agency `942d08d9-82bb-4fd3-9091-ca0c6b50b578`; filters to exactly one date; `CONFIRMED_STATUSES` = BOOKED/BOOKED_BY_AGENT/BOOKED_BY_CUSTOMER/BOOKED_EXTERNALLY/STAY; returns `[]` exit 0 when none; only `--date` is implemented).

## Verification Strategy (MANDATORY)

> **ZERO HUMAN INTERVENTION** — all verification is agent-executed: live Hostfully oracle derivation, live trigger + `task_status_log`, output-vs-oracle comparison, run-log inspection for live-fetch proof, regression diffs, Playwright. No "user visually confirms" criteria.

### The Correctness Definition (judged on OUTPUT)

A produced schedule is CORRECT for a date when, vs the Hostfully-derived oracle: every checkout that date is present and assigned to exactly one correct cleaner per the source-data zone rules; cleaning durations and trash duties match what the live source data implies; per-cleaner and overall totals are internally consistent; correct language/format with real addresses (no placeholder/TODO, no listing/lock codes); nothing omitted, nothing invented. Benign formatting variation allowed; any assignment/duration/trash/totals discrepancy = INCORRECT.

### Live-Fetch Definition (judged on the RUN, not the steps)

The employee FETCHES LIVE when, inspecting the task's run log: it invokes `NOTION_GET_PAGE_MARKDOWN` (or a Notion query) for the source pages at runtime, parses the returned content, and uses those parsed values — AND the `execution_steps` contain no hardcoded times/zones/calendar values. Confirmed by a **causal-mutation test**: change a value in a Notion page (or use a controlled fixture/second read) and observe the output change accordingly; the prior hardcoded employee (Baseline-A) would NOT change.

### No-Plumbing-Leak Definition (judged on the generated archetype text)

The generated `execution_steps`/`delivery_steps` contain none of: `printenv`, `node -e`, `getUTCDay`, `tsx /tools/`, `submit-output`, `<approved-content>`, `/tmp/`. Objective grep gate over the stored archetype fields.

### Test Decision

- **Infrastructure exists**: YES (Vitest). **Automated tests**: tests-after for generator/compiler LOGIC changes (add/adjust unit tests, including the two-prompt-path parity test and the hardcoding-driver grep gate). Prompt/base-layer-only changes validated via Baseline-B regen + live output correctness + run-log inspection.
- **Primary verification = agent-executed**: Hostfully-oracle vs live employee output across 5 dates, every iteration; run-log live-fetch proof; Baseline-B regression diff.

### QA Policy

Evidence to `.sisyphus/evidence2/task-{N}-{slug}.{ext}`.

- **Wizard/clarify chat**: Playwright (real browser, authenticated) at `localhost:7700/dashboard`.
- **Trigger/lifecycle**: Bash (admin trigger) + `task_status_log` polling + Inngest UI + run-log inspection.
- **Output correctness**: Bash/agent reasoning comparing posted schedule to the per-date Hostfully oracle.
- **Live-fetch + no-leak**: run-log grep for Notion calls; archetype-field grep for forbidden plumbing tokens; causal-mutation test.
- **Regression**: Bash + DB (`archetype_generation_calls`, generated archetypes) regen of Baseline-B + the `daily-motivation` genericity probe.

### Oracle-First Rule (CRITICAL)

The per-date oracle MUST be derived in Wave 1 by the agent calling the Hostfully API itself, BEFORE any employee output is judged, and independently of any existing employee. Reverse-justifying the oracle from an employee's output is forbidden. The 3 Notion pages are snapshotted in Wave 1 so the oracle and judgments use identical source data and drift is detectable.
