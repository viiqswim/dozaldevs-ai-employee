# Platform Diagnosis — Cleaning Schedule Employee Defects

## Summary

**12 defects diagnosed | 7 platform fixes designed | 2 defects classified as infrastructure/non-generator | 3 defects are cascaded from root defects**

All 12 observed defects are addressed below. 7 fixes target `src/gateway/services/prompts/archetype-generator-prompts.ts` (hot-reload capable). 0 fixes require changes to harness, delivery phase, worker tools, or other employees.

### Defect 1: DATE BLINDNESS (Critical)

**Observed**: All 3 triggered dates (2026-06-15, 2026-06-20, 2026-06-22) produced the identical June 17 schedule. The trigger body `{"date":"2026-06-20"}` was passed but completely ignored. Execution used "today" (the actual current date at run time) throughout.

**Root Cause**: `SYSTEM_PROMPT_PRE` — `## Input Detection (CRITICAL)` section — does not require that date-parameterized employees detect the date as a runtime input. The rule says "Carefully read the description and identify any values the user must supply at runtime" — but the description "Every morning, I need an employee to check which properties have guests checking out **that day**" is interpreted by the LLM as "this runs on a schedule so I use the system date," rather than "the trigger body supplies the target date." The `## Template Syntax in execution_steps` section defines `{{key}}` but provides no example using a date; the generated employee used literal "today" instead.

Additionally, the existing working archetype (00000000-0000-0000-0000-000000000019) uses `printenv INPUT_DATE` in Step 1 and says "CRITICAL: Use targetDate for ALL steps below. Do NOT use today's date." — but this pattern is not in SYSTEM_PROMPT_PRE and is therefore not generated by the wizard.

**Generic Fix**: Add a mandatory rule to `## Input Detection (CRITICAL)` in `SYSTEM_PROMPT_PRE`:

> **DATE/PERIOD RULE**: When the description implies the employee operates on a specific date, reporting period, or time range that may differ from the actual run date (e.g., "that day", "for that date", "for the period", "for yesterday"), you MUST:
>
> 1. Create an `input_schema` item: `{"key": "target_date", "label": "Target Date", "type": "date", "frequency": "every_run", "required": true, "description": "The date to process. Injected as INPUT_TARGET_DATE env var."}`
> 2. Use `{{target_date}}` in execution_steps wherever the date is referenced.
> 3. The FIRST numbered step in execution_steps MUST read the date: "Read the target date: run `printenv INPUT_TARGET_DATE` and save it as `targetDate`. CRITICAL: Use `targetDate` for every subsequent step. NEVER use the system date or 'today'."

**Fix Target**: `src/gateway/services/prompts/archetype-generator-prompts.ts` — `SYSTEM_PROMPT_PRE`, `## Input Detection (CRITICAL)` section

**Priority**: Critical

### Defect 2: SINGLE CLEANER — Missing Multi-Source Reasoning (Critical)

**Observed**: All schedules assigned only Yessica. Oracle expects 3–4 cleaners per date (Zenaida, Diana, Berenice, plus Yessica depending on the date). The generated execution_steps pulled checkouts from Hostfully but never read Notion source docs (Manual de Personal, Directorio Operativo) to determine cleaner assignments.

**Root Cause**: `SYSTEM_PROMPT_PRE` — `## execution_steps Field Rules (CRITICAL)` section — has no rule requiring a step per data source when multiple are mentioned. The description said "My team uses Notion to track which cleaners cover each area" — the generator correctly identified Notion as relevant in the prose, but execution_steps only included a Hostfully get-checkouts step and skipped the Notion lookup entirely. There is no rule like "each data source mentioned in the description MUST get its own step, with explicit instructions on what to extract and how to cross-reference with other data."

**Generic Fix**: Add a new section `## Multi-Source Reasoning (MANDATORY)` to `SYSTEM_PROMPT_PRE`:

> When the description mentions multiple distinct data sources (e.g., "we use [System A] for X and [System B] for Y"), execution_steps MUST include a dedicated numbered step for EACH data source. The steps must follow this pattern:
>
> 1. Fetch primary data from System A (e.g., checkouts, orders, tickets)
> 2. Fetch reference/lookup data from System B (e.g., team assignments, pricing, config)
> 3. Fetch additional lookup data from System C if mentioned
> 4. Cross-reference: for each item from step 1, apply the rules/assignments from steps 2–3
> 5. Handle cases where no match is found (e.g., mark as UNASSIGNED, use default)
>
> NEVER skip a data source that the description explicitly mentions. If the description says "use Notion to track which cleaners cover each area," there MUST be a step that reads from Notion and a step that applies the zone/assignment rules.

**Fix Target**: `src/gateway/services/prompts/archetype-generator-prompts.ts` — `SYSTEM_PROMPT_PRE`, new section after `## execution_steps Runtime Patterns (MANDATORY)`

**Priority**: Critical

### Defect 4: ENGLISH ONLY — Language Not Specified (High)

**Observed**: All 3 schedules were produced in English. Oracle expects Spanish (the team is Spanish-speaking; the Notion source docs are in Spanish; "Buenos días equipo" style messages are expected).

**Root Cause**: `SYSTEM_PROMPT_PRE` — `## Rules (CRITICAL — never violate)` — the `identity` rule says the identity must include "communication style" but doesn't require language specification. The LLM defaulted to English because no language was specified in the description and no rule forces the generator to infer it.

The description said "create a cleaning schedule for my team. My team uses Notion to track..." — no explicit language mention, but:

- The Notion source docs (connected to the tenant) are in Spanish
- Spanish-speaking cleaning crews are a common context in VLRE's operational setup
- A generic fix can address any employee where non-English output is implied

**Generic Fix**: Add to the `identity` rule in `SYSTEM_PROMPT_PRE`:

> - If the description implies non-English output (e.g., mentions a non-English team, non-English documents, non-English channel names, or the tenant's connected systems have non-English content), the identity MUST specify the output language explicitly. Example: "You produce all schedules and communications in Spanish." If the output language is ambiguous, include a step in execution_steps to ask/infer from the source data.

**Fix Target**: `src/gateway/services/prompts/archetype-generator-prompts.ts` — `SYSTEM_PROMPT_PRE`, `## Rules (CRITICAL)` identity rule

**Priority**: High

### Defect 6: DELIVERY AS THREAD REPLY — Schedules Posted in Thread (Medium)

**Observed**: Cleaning schedules were posted as thread replies to the "Task received" notification message, not as standalone channel posts. Oracle expects a standalone channel post that cleaners can see directly.

**Root Cause**: `SYSTEM_PROMPT_PRE` — `### Template A: Slack delivery` — the delivery step says "Post the approved content to the `$NOTIFICATION_CHANNEL` Slack channel" without specifying standalone vs thread. The platform's post-message.ts tool may inherit a thread context if `$NOTIFY_MSG_TS` is available. The delivery container may be posting in the notification thread because the delivery prompt includes the notification thread timestamp.

**Generic Fix**: Update `### Template A: Slack delivery` in `SYSTEM_PROMPT_PRE` to distinguish between thread-reply delivery and standalone channel delivery:

> - If the deliverable is a standalone announcement (schedule, digest, report meant for a general audience), the delivery step MUST NOT include a `--thread-ts` flag. Standalone posts go to the channel root, not as replies.
> - If the deliverable is a reply to a specific conversation (guest message reply, follow-up), the delivery step SHOULD include `--thread-ts` with the relevant thread timestamp from metadata.
> - Default for NEW-AUDIENCE deliverables (schedules, summaries, reports): omit `--thread-ts` to ensure standalone posting.

**Fix Target**: `src/gateway/services/prompts/archetype-generator-prompts.ts` — `SYSTEM_PROMPT_PRE`, `### Template A: Slack delivery`

**Priority**: Medium

### Defect 9: COMPOSIO TOOL MISSING FROM REGISTRY (High)

**Observed**: The generated execution_steps for Run B referenced "Notion (via Composio)" but `/tools/composio/execute.ts` was NOT in `tool_registry.tools`. The 4 registered tools were: `get-checkouts.ts`, `get-property.ts`, `post-message.ts`, `submit-output.ts`.

**Root Cause**: Contradiction between two sections of `SYSTEM_PROMPT_PRE`:

1. `buildConnectedAppsBlock()` tells the LLM: "Describe actions using these apps in plain English — the runtime Composio skill resolves the exact invocation." This instructs the LLM NOT to use specific tool paths for Composio actions.

2. `SYSTEM_PROMPT_POST` says: "For tool_registry.tools: List tools ONLY from the 'Available Tools' section." The LLM correctly avoids inventing Composio paths, but also fails to include `/tools/composio/execute.ts` (the runtime resolver) in the registry.

Result: The employee's steps reference Composio in prose, but the tool registry doesn't list the composio tool — meaning `enforce_tool_registry` enforcement would block all Composio calls, and human reviewers see an inconsistent archetype.

**Generic Fix**: Add a rule to `buildConnectedAppsBlock()` or to `SYSTEM_PROMPT_PRE`:

> **Composio Tool Registry Rule**: When execution_steps describe any action using a connected Composio app (e.g., "read the Notion page", "add a row to Google Sheets"), you MUST add `/tools/composio/execute.ts` to `tool_registry.tools`. This is the runtime tool that executes ALL Composio app actions — it must always be registered when any Composio-connected app is used.

**Fix Target**: `src/gateway/services/prompts/archetype-generator-prompts.ts` — `buildConnectedAppsBlock()` function, or `SYSTEM_PROMPT_POST` tool_registry notes section

**Priority**: High

### Defect 10: TRIGGER INCONSISTENCY — overview.trigger contradicts trigger_sources.type (Medium)

**Observed**: The generated archetype had `trigger_sources.type = 'manual'` (structured field) but `overview.trigger = 'Scheduled daily at 8 AM'` (prose field). These contradict each other — the employee would never auto-run on a schedule because the structured field says manual, yet the description promises a daily schedule.

**Root Cause**: `SYSTEM_PROMPT_PRE` has no rule enforcing consistency between `overview.trigger` and `trigger_sources.type`. These fields are defined independently in `SYSTEM_PROMPT_POST` — the LLM is free to set them to contradictory values. The generator correctly identified this is a scheduled employee (prose) but failed to propagate that to `trigger_sources.type = 'scheduled'`.

**Generic Fix**: Add a consistency rule to `SYSTEM_PROMPT_PRE` (or to `SYSTEM_PROMPT_POST` notes):

> **Trigger Consistency Rule (MANDATORY)**: `overview.trigger` MUST accurately reflect `trigger_sources.type`:
>
> - `type: 'manual'` → overview.trigger MUST say "Triggered manually by the team" (or equivalent). NEVER describe a schedule or webhook in overview.trigger when type is 'manual'.
> - `type: 'scheduled'` → overview.trigger MUST say "Scheduled: [cron expression or human-readable time]".
> - `type: 'webhook'` → overview.trigger MUST say "Triggered by [event_type] webhook event".
>   These two fields MUST NOT contradict each other. If the description says "every morning", set `trigger_sources.type = 'scheduled'` AND set `overview.trigger = 'Scheduled daily at 8 AM'`.

**Fix Target**: `src/gateway/services/prompts/archetype-generator-prompts.ts` — `SYSTEM_PROMPT_PRE`, new rule in `## Rules (CRITICAL — never violate)` section

**Priority**: Medium

### Defect 11: CLARIFY GATE SKIPPED — Single Sentence Generated Proposal (Critical)

**Observed**: Two test descriptions jumped immediately to `kind:'proposal'` without asking a single clarifying question:

- Run A: "Help me tell my cleaning crew which houses to clean each day." (1 sentence, 80 chars)
- Run B: "Every morning, I need an employee to check which properties have guests checking out that day and create a cleaning schedule for my team. My team uses Notion to track which cleaners cover each area and how long each property takes. The final schedule should be posted to our Slack channel so cleaners know what to do." (3 sentences, 317 chars)

Neither prompted a clarifying question. 0 of 7 critical ambiguities were surfaced (trigger type, input date format, output language, cleaner source, Notion page IDs, ZIP zone rules, trash schedule rules).

**Root Cause**: `buildConverseSystemPromptPre(isCreate=true)` — the converse system prompt has this rule: "Ask a clarifying question ONLY when the request is genuinely ambiguous about WHICH field or WHAT value to change. Prefer acting over asking."

This rule was designed for the EDIT path (modifying an existing archetype where the user has a specific change in mind). On the CREATE path, when the current config is empty (`role_name = ''`), the same "prefer acting over asking" rule causes the LLM to generate a complete proposal from a brief description rather than surfacing critical unknowns.

Additionally, `buildConnectedAppsBlock()` passes connected toolkits (Hostfully, Notion, Slack) to the prompt — the LLM sees all three connected and confidently generates a proposal mapping the description to those tools, skipping clarification entirely.

**Generic Fix**: Add a CREATE-specific rule to `buildConverseSystemPromptPre(isCreate: boolean)` — in the block that runs only when `isCreate = true`:

> **NEW EMPLOYEE CREATION — MANDATORY CLARIFY RULE**: When the current configuration is empty (this is a new employee, not an edit), you MUST ask a clarifying question on the first user turn UNLESS the user's message is at least 200 words AND explicitly states the trigger type, output format, and primary data sources.
>
> Short descriptions (under 200 words) CANNOT contain all the information needed to generate a correct archetype. You MUST ask about the single most critical missing piece:
>
> - If trigger type is unclear (when does this run?): ask about schedule, webhook, or manual trigger
> - If date/period handling is unclear (does it operate on a specific date?): ask if the date varies per run
> - If output language is unclear: ask what language the output should be in
> - If multiple data sources are mentioned but details are missing: ask for specifics
>
> ONE question per turn. Pick the most critical unknown. Do NOT skip this step when connected apps match the description — connected apps reduce the question count but do NOT eliminate the first clarifying question for brief descriptions.

**Fix Target**: `src/gateway/services/prompts/archetype-generator-prompts.ts` — `buildConverseSystemPromptPre()` function, `isCreate = true` branch

**Priority**: Critical

### Defect 12: WRONG MODEL — Generator Proposed minimax/minimax-m2.7 (Non-Generator)

**Observed**: The wizard proposed `minimax/minimax-m2.7` instead of `deepseek/deepseek-v4-flash`. The archetype was later saved with `deepseek/deepseek-v4-flash` (manually patched during Task 10 execution).

**Root Cause**: The model recommendation engine (`src/lib/model-selection/`) correctly ran via `applyModelAndEstimate()` in `converse()` and selected `minimax/minimax-m2.7` based on the archetype profile. This is the engine's legitimate recommendation for this type of employee. The `SYSTEM_PROMPT_PRE` correctly notes that `minimax/minimax-m2.7` is "a default placeholder — the recommendation engine will override this" — the engine overrode it with minimax (its recommendation).

The archetype was saved with `deepseek/deepseek-v4-flash` because the Sisyphus plan executor manually patched it after creation. The runtime tasks (all 3 triggered dates) ran with `deepseek/deepseek-v4-flash` — so the wrong model was NOT the cause of any output defect.

**Generic Fix**: **NOT a generator prompt defect.** The recommendation engine is functioning as designed. Changing which model it recommends requires modifying `src/lib/model-selection/` which is outside scope per task constraints. The wizard's proposal-then-confirm flow allows users to override the recommended model before saving.

**Fix Target**: None — model selection engine outside scope; runtime model was correct

**Priority**: N/A (non-generator, confirmed correct at runtime)
