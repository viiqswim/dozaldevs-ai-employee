## Background: How Supersede Works

When a new webhook arrives for a thread that already has a task in `Reviewing` state, the lifecycle's `check-supersede` step detects the conflict. It:

1. Updates the old task's Slack card to show `⏭️ Superseded`
2. Cancels the old task (status → `Cancelled`)
3. Allows the new task to continue through the normal `Reviewing → approval` flow

The `conversation_ref` used for supersede detection is the Hostfully `thread_uid`.


## ✅ Test Passed

- Task 1: `Cancelled`, Slack card shows `⏭️ Superseded`
- Task 2: `Done`, message delivered to guest
- Only Task 2's response appears in Hostfully — Task 1's draft was never sent

**If Task 1 was NOT cancelled** (still in `Reviewing` after Task 2 reached `Reviewing`), the `check-supersede` step is not detecting the conflict. Check:

- That both tasks share the same `thread_uid` (= `conversation_ref`)
- The `check-supersede` step logic in `src/inngest/employee-lifecycle.ts`
- Inngest run details for Task 2 — did `check-supersede` run and what did it find?

**If both Slack cards still show approval buttons**, the supersede Slack update failed. Check gateway logs for errors when updating the old Slack message.
