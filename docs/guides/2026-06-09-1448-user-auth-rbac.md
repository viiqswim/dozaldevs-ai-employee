# User Auth & RBAC Guide

Multi-tenant user authentication and role-based access control for the AI Employee Platform.

---

## Overview

The platform uses **Supabase Auth** for identity (JWT issuance) and a custom **Express middleware chain** for authorization. Two token types are accepted on all `/admin/*` and `/me` endpoints:

| Token type        | Value                                          | Use case                                          |
| ----------------- | ---------------------------------------------- | ------------------------------------------------- |
| **SERVICE_TOKEN** | Opaque hex string from `SERVICE_TOKEN` env var | External cron callers, scripts, Inngest functions |
| **Supabase JWT**  | Short-lived JWT issued by Supabase Auth        | Dashboard users, logged-in humans                 |

`ADMIN_API_KEY` / `X-Admin-Key` are **removed** â€” not deprecated, gone since T24. All callers must use `Authorization: Bearer`.

---

## Auth Middleware Resolution Order

`authMiddleware` in `src/gateway/middleware/auth.ts` resolves identity in this order:

1. **SERVICE_TOKEN** â€” timing-safe compare of the Bearer token against `SERVICE_TOKEN()`. Sets `req.isServiceToken = true`. Bypasses all membership checks.
2. **Supabase JWT** â€” `verifySupabaseJwt(token)` + `ensureUserExists(claims)` upsert. Sets `req.auth` to the `AuthenticatedUser`. Returns 403 `ACCOUNT_DISABLED` if `user.status !== 'active'`.
3. **No match** â€” 401 `AUTHENTICATION_REQUIRED`.

## Dual-Env JWT Profiles

The gateway detects its profile at startup via `detectEnvProfile()` in `src/lib/config.ts`:

| Profile   | Detection                                                                               | JWT algorithm | Verification                              |
| --------- | --------------------------------------------------------------------------------------- | ------------- | ----------------------------------------- |
| **LOCAL** | `SUPABASE_URL` starts with `http://localhost` and `SUPABASE_ANON_KEY` starts with `eyJ` | HS256         | Symmetric secret from `GOTRUE_JWT_SECRET` |
| **CLOUD** | `SUPABASE_URL` is `https://*.supabase.co` and `SUPABASE_ANON_KEY` starts with `sb_`     | ES256         | Asymmetric JWKS from `SUPABASE_JWKS_URL`  |

Mixing LOCAL and CLOUD values causes a fatal error at startup. `SUPABASE_JWKS_URL` is derived automatically: `${SUPABASE_URL}/auth/v1/.well-known/jwks.json`.

**Local dev default**: `GOTRUE_JWT_SECRET` defaults to `super-secret-jwt-token-with-at-least-32-characters-long` in the Docker Compose auth container â€” the gateway uses the same fallback, so local JWTs verify without setting `GOTRUE_JWT_SECRET` in `.env`.

## Supabase Key Model

The platform uses the new Supabase opaque key model:

| Key type         | Env var               | Format               | Use                                                                                                                                                     |
| ---------------- | --------------------- | -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Publishable**  | `SUPABASE_ANON_KEY`   | `sb_publishable_...` | Safe for browser use. Used as `apikey` header for PostgREST and Auth calls (login, signup).                                                             |
| **Secret**       | `SUPABASE_SECRET_KEY` | `sb_secret_...`      | Server-side only. Used as both `apikey` and `Authorization: Bearer` for admin Auth API calls (user creation, invitations). **Never expose to browser.** |
| **Legacy HS256** | `SUPABASE_ANON_KEY`   | `eyJ...` (JWT)       | LOCAL profile only. Still valid for local dev â€” the `eyJ` prefix triggers LOCAL profile detection.                                                      |

**Critical**: opaque `sb_secret_*` keys are NOT JWTs â€” they cannot be used as `Authorization: Bearer` tokens in PostgREST calls. For server-side PostgREST reads, use `apikey: SUPABASE_SECRET_KEY()` only (no Authorization header).

## Authorization Middleware

`src/gateway/middleware/authz.ts` exports three guards:

- `requireAuth` â€” plain middleware; passes if `req.isServiceToken` or `req.auth` is set. Returns 401 otherwise.
- `requireTenantRole(...roles)` â€” factory; checks the user's `TenantMembership` for the `:tenantId` route param. SERVICE_TOKEN and PLATFORM_OWNER bypass the membership check. Returns 403 if the user's role rank is below the minimum required.
- `requirePermission(permission)` â€” factory; checks `ROLE_PERMISSIONS` or `TENANT_ROLE_PERMISSIONS` for the named permission. SERVICE_TOKEN and PLATFORM_OWNER always pass.

Role rank order (highest to lowest): `OWNER(4) > ADMIN(3) > MEMBER(2) > VIEWER(1)`.

**Standard guard chain for a tenant-scoped route:**

```typescript
router.get(
  '/admin/tenants/:tenantId/things',
  authMiddleware,
  requireAuth,
  requireTenantRole(TenantRole.VIEWER),
  async (req, res) => { ... }
);
```

## RBAC â€” Roles and Permissions

### Global Roles (`Role` enum, `src/lib/auth/permissions.ts`)

| Role             | Scope                   | Key permissions                                                                  |
| ---------------- | ----------------------- | -------------------------------------------------------------------------------- |
| `PLATFORM_OWNER` | Cross-tenant superadmin | All permissions                                                                  |
| `ADMIN`          | Platform-level          | Manage archetypes, rules, KB, locks, projects, trigger employees, invite members |
| `EDITOR`         | Platform-level          | Manage archetypes, rules, KB (no trigger)                                        |
| `USER`           | Platform-level          | Trigger employees, read tasks                                                    |
| `VIEWER`         | Platform-level          | Read tenant and tasks only                                                       |

### Tenant Roles (`TenantRole` enum)

| Role     | Key permissions                                                                          |
| -------- | ---------------------------------------------------------------------------------------- |
| `OWNER`  | All tenant permissions including delete tenant, manage secrets/integrations/members      |
| `ADMIN`  | Manage archetypes, rules, KB, locks, projects, trigger, invite (no secrets/integrations) |
| `MEMBER` | Trigger employees, read tasks                                                            |
| `VIEWER` | Read tenant and tasks only                                                               |

## Database Models

### `users` table

`id, supabase_id (unique, nullable), email, name, role (Role enum), status, created_at, updated_at, deleted_at`

Created/updated via `ensureUserExists()` on every authenticated request. `status = 'disabled'` is the immediate lockout mechanism â€” checked per-request in `authMiddleware`.

### `tenant_memberships` table

Composite PK `[tenant_id, user_id]`. Fields: `tenant_id, user_id, role (TenantRole enum), joined_at, deleted_at`. Soft-delete only. Scoped by `tenant_id` on every query.

### `tenant_invitations` table

`id, tenant_id, email, role (TenantRole), token (unique), status, expires_at, accepted_at, declined_at, revoked_at, inviter_id, created_at`. No `deleted_at` â€” status transitions (`pending â†’ accepted/declined/revoked`) are the lifecycle. Token is a 32-byte random hex string; expires in 7 days.

---

## Invitation Flow

1. **Create** â€” `POST /admin/tenants/:tenantId/invitations` (requires ADMIN or OWNER role). Creates a `TenantInvitation` row and provisions the invitee in Supabase Auth (magic link / invite email).
2. **Accept** â€” `POST /invitations/accept` with `{ token }` in the request body. No auth required â€” user may not be authenticated when clicking the magic link. Creates a `TenantMembership` row. Returns 410 if already used.
3. **Decline** â€” `POST /invitations/decline` with `{ token }`. No auth required.
4. **Revoke** â€” `POST /admin/tenants/:tenantId/invitations/:id/revoke` (requires ADMIN or OWNER role). Sets `status = 'revoked'`.

**Note**: Accept/decline endpoints use `/invitations/accept` and `/invitations/decline` with `{ token }` in the request body â€” NOT `/invitations/:token/accept`.

## User Deactivation

Set `users.status = 'disabled'` via `PATCH /admin/tenants/:tenantId/members/:userId` (or direct DB update). The `authMiddleware` checks `user.status` on every request â€” the lockout is immediate, no token invalidation needed.

The Supabase Admin API ban (`PUT /auth/v1/admin/users/:supabase_id` with `ban_duration`) is belt-and-suspenders for blocking token refresh. The app-level `status` check is the primary mechanism.

## Known Gotchas

1. **`python3` is blocked by asdf** in this repo (no version in `.tool-versions`). For JWT header decoding in shell scripts, use `echo "$JWT" | cut -d. -f1 | base64 -d` instead.

2. **Kong port is 54331, not 54321** in the local Docker setup. `docker port ai-employee-kong` â†’ `8000â†’54331`. Auth = `http://localhost:54331/auth/v1`, REST = `http://localhost:54331/rest/v1`.

3. **`docker compose restart` does NOT re-read env vars** â€” use `docker compose up -d --force-recreate auth` after changing auth container env vars.

4. **Stale Prisma LSP errors** â€” `TenantRole`, `Role`, `prisma.user`, `prisma.tenantMembership` may show as missing in the LSP. Run `pnpm prisma generate` and use `pnpm build` (tsc) as ground truth.

5. **`ban_duration: 'none'` is a no-op** â€” it does NOT ban the user in Supabase Auth. Use `ban_duration: '876000h'` (100 years) for an effective ban. The app `status='disabled'` check is the real lockout.

6. **Cloud DB schema drift** â€” if `prisma migrate status` says "up to date" but columns are missing, the migrations were manually inserted into `_prisma_migrations` without their DDL running. Apply the missing `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` statements via psql, then `NOTIFY pgrst, 'reload schema'`.

7. **`ensureUserExists` must run before seeding memberships** â€” hitting `/me` with a fresh Supabase JWT auto-creates the `users` row (upsert by `supabase_id = sub`). The `tenant_memberships` FK to `users.id` requires the user row to exist first.

8. **Authz fires before the query** â€” `requireTenantRole` rejects non-members before any Prisma call. Cross-tenant isolation has two independent layers: (a) membership gate â†’ 403 for non-members, (b) `where: { tenant_id }` â†’ row scoping for members who pass the gate.
