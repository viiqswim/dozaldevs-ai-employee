# Information Drift Audit — AI Employee Platform

## 1. Purpose

The platform encodes the same conceptual facts (output-contract paths, classification enums, the execution prompt, phase values, tool CLI docs, skill metadata, env-var requirements) in many independent places. When one copy changes, the others silently drift, producing: invalid generated archetypes, stale tool docs read by live agents, employees told about tools/skills not in their container, and — in one verified case — a production wizard that silently loses its tool catalog.

This document maps every drift point with evidence, names the intended single source per fact, and lists the intentional asymmetries that must NOT be "consolidated."

## 2. The Three Module Worlds (critical constraint)

Consolidation is bounded by a hard module-resolution fact:

| World | Members                       | Compiled?                                                                         | Can import `src/lib/`?                                     |
| ----- | ----------------------------- | --------------------------------------------------------------------------------- | ---------------------------------------------------------- |
| **A** | gateway `.ts` + worker `.mts` | Yes (`tsconfig.build.json`, → `dist/`)                                            | YES (precedent: `src/lib/go-models.ts`, imported as `.js`) |
| **B** | `src/worker-tools/**/*.ts`    | No — run under `tsx`, own `node_modules`, **excluded from `tsconfig.build.json`** | **NO**                                                     |

Consequence: a `src/lib/` constant cannot reach World B. Facts shared across both worlds (notably `/tmp/summary.txt`) require a **bridge**: a duplicate constant in `src/worker-tools/lib/` plus a sync test asserting equality with the canonical `src/lib/` value.

## 3. Drift Map by Area

### Area 1 — Tools

**Authoritative source**: the tool scripts in `src/worker-tools/**/*.ts` (`--help` strings, `requireEnv()` calls, JSDoc). Docker `COPY ... /tools/` is wholesale (low drift).

**Good existing pattern**: `src/gateway/services/tool-parser.ts` `discoverTools(basePath)` (line 65) regex-derives flags/env/output from source. Consumed by `archetype-generator.ts:127` and `admin-tools.ts`.

| Duplicate                 | Location                                                                                          | Drift risk                                                            |
| ------------------------- | ------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- |
| Full CLI re-documentation | `src/workers/skills/tool-usage-reference/SKILL.md` (~1034 lines, hand-written, no sync to source) | **HIGH** — read by every agent at runtime; stale flags → agent errors |
| Per-employee tool paths   | `prisma/seed.ts` `tool_registry.tools[]` (no existence validation)                                | **HIGH** — renamed tool silently breaks registry                      |
| Service directory table   | `AGENTS.md` (Shell tools table)                                                                   | LOW (rarely changes)                                                  |
| Hardcoded tool rules      | `archetype-generator-prompts.ts` ("always include submit-output", github get-token)               | MEDIUM                                                                |

> **Note**: `tool_registry` is **metadata-only** — `execution-phase.mts` reads it but does NOT restrict tool access. All `/tools/` are always available. Validation (path existence) is appropriate; runtime enforcement is explicitly out of scope.

### Area 2 — Skills

## 6. PRODUCTION FINDING (latent bug surfaced during audit)

`archetype-generator.ts:126` resolves the tool catalog at runtime via:

```
const basePath = path.join(process.cwd(), 'src/worker-tools');
const tools = await discoverTools(basePath);
```

But `Dockerfile.gateway` (lines 77-89) copies only `dist/`, `dashboard/dist`, `prisma/`, `package.json`, `node_modules` into the runtime image — it does **NOT** copy `src/worker-tools/`. In production the `readdir` fails, the catch at `archetype-generator.ts:145` logs `discoverTools failed — falling back to base system prompt without tool catalog`, and the wizard generates archetypes **without any tool catalog**.

This is a real, separate latent bug. The consolidation must at minimum surface it. Recommended handling (in plan Task 9): verify the Dockerfile.gateway COPY and add `COPY --from=builder /build/src/worker-tools ./src/worker-tools` (source is needed at runtime), OR change the generator to read from a generated JSON manifest that IS in `dist/`. Flagged to F1 for an explicit decision; do NOT silently ignore.

## 8. Intentional Asymmetries — DO NOT Consolidate

1. `EMPLOYEE_PHASE` (lifecycle→harness router) ≠ `TASK_PHASE` (worker→Composio audit). Different variables.
2. `localCriticalVars` vs `flyCriticalVars` — the 2-key delta is correct (fly injects Inngest vars explicitly).
3. Delivery `compileAgentsMd` passes empty `employeeRules`/`employeeKnowledge` deliberately.
4. The two env builders (execution `machine-provisioner.ts` vs delivery `delivery-retry.ts`) — separate by design.
5. Static `src/workers/config/opencode.json` — a fallback safety net, NOT dead code.
6. `submitOutputCmd` execution variant is dynamic (`approvalRequired ? ...`); delivery is always `NO_ACTION_NEEDED`.
7. `postgrest-client.ts` raw `process.env` usage — AGENTS.md forbids "fixing" it.
8. Clusters B/D/F (wording-sensitive prompt text) — left inline in this plan.

## 9. Byte-Identity Guarantee (the safety net)

There is no byte-identity anchor today (the only generator test is a reference comparison). Before ANY extraction, golden fixtures capture current output of:

- the archetype-generator system prompt builder (empty toolkits)
- `REFINE_SYSTEM_PROMPT`
- `compileAgentsMd()` for a fixed representative archetype input

Every subsequent wave re-runs `pnpm test:unit -- golden-prompts` and must stay GREEN. A diff = the refactor changed behavior = fail. This converts a risky prompt-touching refactor into a provably behavior-preserving one.

## 10. Resolved Items

### Delivery field consolidation (enforce-execution-delivery-phases plan, June 2026) — RESOLVED

**Problem**: Two delivery fields existed on the `archetypes` table (`delivery_steps` and `delivery_instructions`), with three lifecycle gate sites each implementing their own fallback logic (`delivery_steps ?? delivery_instructions ?? ''`). This created drift risk and made it unclear which field was authoritative.

**Resolution**:

- `delivery_instructions` column dropped via Prisma migration `20260616030000_consolidate_delivery_fields`
- `delivery_steps` is now the single canonical delivery field
- `src/lib/delivery-resolver.ts` created — exports `resolveDelivery(archetype, classification)` returning a `DeliveryResolution` discriminated union (`has-delivery | no-delivery-escape-hatch | misconfigured`)
- All 3 lifecycle gate sites (`no-approval-path.ts`, `delivery-retry.ts`, `approval-handler.ts`) now call `resolveDelivery()` — no inline fallback logic
- Hard gate added at POST/PATCH `/archetypes`: `deliverable_type` set + empty `delivery_steps` returns `400 MISSING_DELIVERY_CONFIG`
- Escape hatch preserved: `deliverable_type: null` + `delivery_steps: null` is valid (employee delivers inside execution and emits `NO_ACTION_NEEDED`)
- `delivery-resolver.ts` retains a `delivery_instructions` fallback field in its interface for transition tolerance (existing DB rows with the old field populated still work), but the column is gone from the schema
