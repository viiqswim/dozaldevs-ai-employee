# Email Setup — Mailpit (Local) and Resend (Production)

The platform sends transactional emails for one purpose: **tenant invitations**. The email system is environment-aware — it uses a local SMTP trap (Mailpit) during development and the Resend API in production. No code changes are required to switch between them; the provider is selected at startup based on a single env var.

---

## Provider Selection

`src/lib/email/index.ts` — `createEmailProvider()` factory, called once at first use via `getEmailProvider()` singleton.

| `RESEND_API_KEY` set? | Provider selected                             | Where emails go              |
| --------------------- | --------------------------------------------- | ---------------------------- |
| No (default)          | `SmtpEmailProvider` (nodemailer)              | Mailpit at `localhost:54325` |
| Yes                   | `ResendEmailProvider` (official `resend` SDK) | Real inboxes via Resend      |

The singleton is created on the **first call to `getEmailProvider()`**. If you add `RESEND_API_KEY` to a running gateway process after it has already sent an email, you must restart the process — the provider will not switch mid-session.

---

## Environment Variables

All four are read via `src/lib/config.ts` accessors:

| Variable             | Default                             | Production value                         |
| -------------------- | ----------------------------------- | ---------------------------------------- |
| `RESEND_API_KEY`     | `""` (empty → use Mailpit)          | Resend API key (`re_...`)                |
| `EMAIL_FROM`         | `DozalDevs <noreply@dozaldevs.com>` | Same (apex domain is verified)           |
| `DASHBOARD_BASE_URL` | `http://localhost:7700`             | `https://ai-employees-laaa.onrender.com` |
| `SMTP_URL`           | `smtp://localhost:54324`            | Not used in production                   |

**`DASHBOARD_BASE_URL` is critical** — it controls the accept-invite link embedded in the email. If unset in production, every invite link points at `localhost:7700` and the recipient cannot accept.

## Local Development — Mailpit

Mailpit is a local SMTP trap included in the Docker Compose stack. It catches all outgoing email and makes it inspectable in a web UI — nothing actually leaves your machine.

- **SMTP port**: `54324` (gateway sends here)
- **Web UI**: `http://localhost:54325` — open this to read caught emails, including invitation links
- **Start**: `pnpm docker:start` (Mailpit runs as part of the shared Docker Compose)
- **`RESEND_API_KEY`**: must be absent or empty in `.env` for Mailpit to be used

**Testing an invitation locally:**

1. Trigger `POST /admin/tenants/:tenantId/invitations` (see below)
2. Open `http://localhost:54325` — the email will appear there
3. Copy the accept link from the email body
4. Navigate to the link in your browser to complete the accept flow

## Production — Resend

### Verified domain

The production sending domain is **`dozaldevs.com`** (apex) — verified in Resend. The original plan specified `send.dozaldevs.com` as the subdomain, but the user chose to verify and use the apex domain directly since it was already set up. Both `EMAIL_FROM` and Resend's domain verification use `dozaldevs.com`.

- **Resend dashboard**: https://resend.com/domains — confirms `dozaldevs.com` `sending: enabled`
- **`EMAIL_FROM`**: `DozalDevs <noreply@dozaldevs.com>`

### Verifying a send

After triggering an invitation in production, confirm delivery via the Resend API:

```bash
curl -s -H "Authorization: Bearer $RESEND_API_KEY" \
  "https://api.resend.com/emails?limit=5" | jq '.data[] | {id, to, last_event}'
```

`last_event: delivered` = email reached the recipient's mail server. `last_event: bounced` = bad address or domain block.

## Email Template

`src/lib/email/templates/invitation.ts` — `buildInvitationEmail(params)`.

| Field         | Value                                                        |
| ------------- | ------------------------------------------------------------ |
| Subject       | `You've been invited to join {organizationName}`             |
| Accept link   | `{DASHBOARD_BASE_URL}/dashboard/accept-invite?token={token}` |
| Expiry notice | "This invitation expires in 7 days."                         |
| Format        | HTML + plain-text fallback                                   |

The token is a 32-byte random hex string generated at invite-creation time, stored in `tenant_invitations.token`.

## Invitation API

```bash
# Create an invitation (requires ADMIN or OWNER role, or SERVICE_TOKEN)
curl -X POST https://<host>/admin/tenants/<tenantId>/invitations \
  -H "Authorization: Bearer $SERVICE_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"email":"user@example.com","role":"MEMBER"}'
# Response: {id, email, role, status, expiresAt}
# NOTE: the token is NOT in the response — it is only in the email and the DB.
```

To read the token for testing (use service-role key):

```bash
curl -s "$SUPABASE_URL/rest/v1/tenant_invitations?email=eq.<urlenc-email>&status=eq.pending&select=token" \
  -H "apikey: $SUPABASE_SECRET_KEY" \
  -H "Authorization: Bearer $SUPABASE_SECRET_KEY"
```

Accept URL: `{DASHBOARD_BASE_URL}/dashboard/accept-invite?token=<token>`

## Known Gotchas

### 1. POST /invitations does not return the token

The create response contains `{id, email, role, status, expiresAt}` only — no `token` field. The token lives in the email and in `tenant_invitations.token` (PostgREST readable with the service-role key). Plan accordingly when writing E2E tests.

### 2. Transient accept errors (fixed)

The `/invitations/accept` endpoint uses a Prisma Serializable transaction. Under certain race conditions, a PostgreSQL serialization failure (P2034 / error code 40001) previously escaped the JSON error handler and surfaced as a raw HTML 500. This is fixed: the handler now retries up to 3 times on P2034, with 50ms/100ms backoff. If all retries fail, it returns a JSON 409 `SERIALIZATION_FAILURE` instead of a raw HTML 500. In practice a single retry always succeeds.

### 3. Authenticated-but-memberless user after accept failure

If `acceptInvitation` fails (e.g. transient 500, P2034) _after_ `signInWithPassword` already succeeded, the user lands in the dashboard authenticated but with no tenant membership — every protected endpoint returns 403. The fixed `AcceptInvitePage` tracks this with a `signedIn` local variable: if `acceptInvitation` fails after sign-in, the page stays on the accept flow and shows a retry "Accept invitation" button. The `isExistingUser` branch also detects an active Supabase session and renders the Accept button directly instead of the dead "log in" link.

**Recovery for users already stuck in this state** — create the membership and mark the invitation accepted directly:

```bash
# 1. Get the user's internal ID
curl -s "$SUPABASE_URL/rest/v1/users?email=eq.<email>&select=id,supabase_id" \
  -H "apikey: $SUPABASE_SECRET_KEY" -H "Authorization: Bearer $SUPABASE_SECRET_KEY"

# 2. Get the invitation record
curl -s "$SUPABASE_URL/rest/v1/tenant_invitations?email=eq.<email>&status=eq.pending&select=id,tenant_id,role" \
  -H "apikey: $SUPABASE_SECRET_KEY" -H "Authorization: Bearer $SUPABASE_SECRET_KEY"

# 3. Create the membership (replace <...> placeholders)
NOW=$(date -u +"%Y-%m-%dT%H:%M:%S+00:00")
curl -s -X POST "$SUPABASE_URL/rest/v1/tenant_memberships" \
  -H "apikey: $SUPABASE_SECRET_KEY" -H "Authorization: Bearer $SUPABASE_SECRET_KEY" \
  -H "Content-Type: application/json" -H "Prefer: return=representation" \
  -d "{\"tenant_id\":\"<tenant_id>\",\"user_id\":\"<users.id>\",\"role\":\"<ADMIN|MEMBER|VIEWER>\",\"joined_at\":\"$NOW\"}"

# 4. Mark the invitation accepted
curl -s -X PATCH "$SUPABASE_URL/rest/v1/tenant_invitations?id=eq.<invitation_id>" \
  -H "apikey: $SUPABASE_SECRET_KEY" -H "Authorization: Bearer $SUPABASE_SECRET_KEY" \
  -H "Content-Type: application/json" -H "Prefer: return=representation" \
  -d "{\"status\":\"accepted\",\"accepted_at\":\"$NOW\"}"
```

### 4. Resend rejects certain test domains in production

Cloud Supabase Auth (and Resend) rejects `@example.com` and `@test.com` email domains. Use a real domain for E2E production tests (e.g. a `+alias` of your real address).

### 5. Re-inviting a pending address supersedes the old token

If you POST an invitation for an email that already has a `pending` invitation, the prior token is revoked and a fresh token is issued. Only one active token per email per tenant at a time.

### 6. Provider singleton — restart required to switch

The email provider (`ResendEmailProvider` or `SmtpEmailProvider`) is instantiated once on the first email send and cached for the process lifetime. Adding or removing `RESEND_API_KEY` from a running gateway has no effect until restart.
