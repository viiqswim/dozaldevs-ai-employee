# Engineer Employee — Operational Details

> This document is loaded on-demand. For platform-wide rules, see AGENTS.md.

## Engineer Employee

- **Archetype ID**: `ad3531f5-080d-4fd3-a201-5f1c50c67f81` (DozalDevs)
- **Tenant**: DozalDevs (`00000000-0000-0000-0000-000000000002`)
- **role_name**: `engineer` · **model**: `deepseek/deepseek-v4-flash` (recommended) · **approval_required**: true
- **vm_size**: `performance-1x` (required — OpenCode binary reserves ~74GB virtual memory)
- **concurrency_limit**: 1 (only one coding task runs at a time)
- **Trigger**: Manual — dashboard trigger button or admin API with `{ "inputs": { "prompt": "..." } }` body

**What it does**: Receives per-task coding instructions via the trigger prompt, clones the configured GitHub repository, creates a branch (`ai/{taskId}-{slug}`), implements the requested changes, runs tests, creates a pull request, and submits the PR URL for human approval. The lifecycle goes through the full approval path: `Executing → Submitting → Reviewing → Approved → Delivering → Done`.

**Inbound flow**:

```
PM clicks Trigger in dashboard (or calls admin API with prompt)
  → Gateway creates task with raw_event = { inputs: { prompt: "<instructions>" } }
  → inngest.send('employee/task.dispatched')
  → Universal lifecycle: Received → Ready → Executing
  → Worker reads prompt from raw_event.inputs (unwrapped by opencode-harness.mts, injected via trigger-payload.mts as ## Your Assignment)
  → tsx /tools/github/get-token.ts → writes token to /tmp/github-token
  → git clone https://x-access-token:$(cat /tmp/github-token)@github.com/<repo> /tmp/workspace/repo
  → git checkout -b ai/<taskId>-engineer
  → Implements changes per prompt instructions
  → pnpm install --ignore-scripts && npx prisma generate && pnpm build && pnpm test -- --run
  → git commit && git push
  → gh pr create → PR URL captured
  → submit-output.ts --classification NEEDS_APPROVAL --draft-file /tmp/summary.txt
  → Lifecycle: Submitting → Reviewing (PM approves PR in Slack card)
  → Approved → Delivering → Done
```

## Setup

### 1. Connect GitHub via Dashboard

Before creating the employee, connect your GitHub App to the tenant:

1. Go to `http://localhost:7700/dashboard/integrations?tenant=<tenantId>`
2. Click **Connect** next to GitHub
3. Complete the GitHub App installation flow
4. Confirm the "✓ Connected" badge appears

The GitHub App installation ID is stored as a tenant secret (`github_installation_id`). The engineer employee uses this to generate short-lived installation tokens (1-hour TTL) via the internal token endpoint.

### 2. Create via Wizard

1. Go to `http://localhost:7700/dashboard/employees/new?tenant=<tenantId>`
2. Describe the employee in plain English, e.g.: "An engineer that receives coding tasks, implements them in our GitHub repo, runs tests, and creates pull requests for review."
3. The archetype generator auto-detects code-writing employees and sets:
   - `concurrency_limit: 1`
   - `vm_size: 'performance-1x'`
   - `approval_required: true`
   - `worker_env: { GITHUB_REPO_URL: '' }` (fill in the repo picker)
   - `/tools/github/get-token.ts` added to `tool_registry`
4. Select the target repository in the **GitHub Repository** section of the wizard
5. Save as draft, then set `status` to `active`

### 3. Override the Model (Recommended for Testing)

The wizard may select a catalog model that doesn't reliably call bash tools. Override to `deepseek/deepseek-v4-flash` before the first run:

```bash
ARCHETYPE_ID=<archetype_id>
psql postgresql://postgres:postgres@localhost:54322/ai_employee \
  -c "UPDATE archetypes SET model = 'deepseek/deepseek-v4-flash' WHERE id = '$ARCHETYPE_ID';"
```

### Admin API

```bash
source .env
TENANT=00000000-0000-0000-0000-000000000002

curl -s -X POST \
  "http://localhost:7700/admin/tenants/$TENANT/employees/engineer/trigger" \
  -H "X-Admin-Key: $ADMIN_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{"inputs": {"prompt": "Add a health check endpoint at GET /ping that returns {\"status\": \"ok\"}"}}' \
  | jq '{task_id: .task_id, status_url: .status_url}'
```

The `inputs.prompt` field is forwarded to the worker as the `## Your Assignment` section of the initial OpenCode message (via raw_event.inputs unwrapping in the harness).

### Required Tenant Secrets

The GitHub App installation ID must be stored as a tenant secret. This is set automatically when the PM completes the GitHub OAuth flow in the dashboard. To set it manually:

```bash
curl -X PUT "http://localhost:7700/admin/tenants/$TENANT/secrets/github_installation_id" \
  -H "X-Admin-Key: $ADMIN_API_KEY" -H "Content-Type: application/json" \
  -d '{"value":"<installation_id>"}'
```

### Required Environment Variables

These must be set in `.env` (and on Render for production):

```bash
GITHUB_APP_ID=<your-github-app-id>
GITHUB_APP_NAME=<your-github-app-slug>   # URL slug, e.g. "my-ai-employee"
GITHUB_PRIVATE_KEY=<pem-key-with-\n-escaping>
```

The private key is the RSA key downloaded when creating the GitHub App. Store it with literal `\n` characters for single-line env var storage.

## Known Gotchas

**`pnpm install` blocks with `ERR_PNPM_IGNORED_BUILDS`**: pnpm 11 requires explicit build approval for packages like `@prisma/client`, `esbuild`, and `protobufjs`. The workaround is `pnpm install --ignore-scripts` followed by `npx prisma generate`. The execution_steps in the archetype must use this two-step pattern, not a bare `pnpm install`.

**Correct repo URL**: The test target repo is `viiqswim/dozaldevs-ai-employee`, not `dozal-devs/ai-employee`. Using the wrong URL causes a 404 clone failure. The `GITHUB_REPO_URL` worker env var must be set to the full HTTPS URL: `https://github.com/viiqswim/dozaldevs-ai-employee`.

**GitHub token TTL is 1 hour**: Installation tokens expire after 60 minutes. The `get-token.ts` shell tool always fetches a fresh token from the gateway — never cache the token value across steps. Call `tsx /tools/github/get-token.ts` immediately before any git or gh CLI operation that needs auth.

**Only one task at a time**: `concurrency_limit: 1` means a second trigger while a task is `Executing` will queue (or fail, depending on lifecycle config). Don't trigger multiple coding tasks simultaneously — they'd conflict on the same branch namespace.

**`vm_size` is mandatory**: Without `performance-1x`, the Fly.io machine defaults to `shared-cpu-1x` (256MB RAM). The OpenCode binary reserves ~74GB virtual memory at startup and OOM-kills within 45 seconds with 0 tokens generated. Always verify `vm_size` is set in both the DB and the archetype seed.

**Branch naming**: Branches follow `ai/{taskId}-{slug}` where `taskId` is the full UUID and `slug` is the archetype `role_name`. Example: `ai/a1b2c3d4-e5f6-7890-abcd-ef1234567890-engineer`. The `gh pr create` command picks this up automatically.

**Test suite pre-existing failures**: The test repo has 4 pre-existing test failures (`get-properties.test.ts` ×1, `notion/get-page.test.ts` ×3). These are infrastructure tests that require external credentials. The engineer employee should not treat these as regressions — the execution_steps must account for them (e.g. check that no NEW failures were introduced, not that the suite is 100% green).

**`platform_rules_override`**: The archetype generator sets a workspace authorization override for code-writing employees. This replaces the default platform rules section in the compiled AGENTS.md with text that explicitly authorizes the employee to write and push code. If this field is null, the employee may refuse to make changes due to the default read-only platform rules.
