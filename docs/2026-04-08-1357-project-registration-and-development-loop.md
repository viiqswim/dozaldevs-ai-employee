# Registering Projects and Triggering AI Development

This guide is for developers who want to connect a GitHub repository to the AI Employee platform so that Jira tickets automatically become pull requests.

---

## Overview

Before the AI can work on a ticket, the platform needs to know which GitHub repo corresponds to which Jira project. That mapping is a **project registration** — a record stored in the database that links a `jira_project_key` to a `repo_url`, a default branch, and an optional tooling configuration.

When a Jira webhook arrives, the gateway looks up the incoming `issue.fields.project.key` against all registered projects. If it finds a match, it creates a task and kicks off the AI development loop. If it doesn't, the webhook is rejected with `action: project_not_registered`. The `jira_project_key` is the single field that connects an incoming webhook to the right repository.

---

## Prerequisites

Before registering a project or triggering a task, make sure the following are in place:

- **Services running**: `pnpm dev:start` starts the gateway (port 3000) and Inngest Dev Server (port 8288)
- **`ADMIN_API_KEY` in `.env`**: auto-generated by `pnpm setup`; generate manually with `openssl rand -hex 32`
- **`GITHUB_TOKEN` in `.env`**: must have push access to every repo you register — it's a single global token shared across all projects
- **`OPENROUTER_API_KEY` in `.env`**: used by the worker container for AI code generation
- **Docker image built**: `docker build -t ai-employee-worker:latest .` — required before any task can run; rebuild after any change to `src/workers/`
- **Target GitHub repo exists**: the repo at `repo_url` must be accessible by `GITHUB_TOKEN`
- **`PLAN_VERIFIER_MODEL` in `.env`** (optional): Set to `anthropic/claude-haiku-4-5` to enable the plan judge gate, which verifies the AI's plan matches the ticket before coding begins. Leave empty (default) to disable.

### Using curl directly

The equivalent curl command for the same project:

```bash
curl -X POST http://localhost:3000/admin/projects \
  -H "Content-Type: application/json" \
  -H "X-Admin-Key: $ADMIN_API_KEY" \
  -d '{
    "name": "My TypeScript App",
    "repo_url": "https://github.com/acme/my-typescript-app",
    "jira_project_key": "ACME",
    "default_branch": "main",
    "tooling_config": {
      "install": "npm ci",
      "typescript": "npx tsc --noEmit",
      "lint": "npx eslint src",
      "unit": "npx vitest run"
    }
  }'
```

For a minimal registration (name, repo, and key only), `tooling_config` defaults to `pnpm install --frozen-lockfile` for the install step and skips all validation stages:

```bash
curl -X POST http://localhost:3000/admin/projects \
  -H "Content-Type: application/json" \
  -H "X-Admin-Key: $ADMIN_API_KEY" \
  -d '{
    "name": "My TypeScript App",
    "repo_url": "https://github.com/acme/my-typescript-app",
    "jira_project_key": "ACME"
  }'
```

A successful registration returns HTTP 201 with the full project object including the generated `id`.

## The AI Development Loop

### Step 1 — Prepare the webhook payload

The file `test-payloads/jira-realistic-task.json` is the payload `pnpm trigger-task` sends to the gateway. Edit it to describe your ticket:

```json
{
  "issue": {
    "key": "ACME-1",
    "fields": {
      "project": { "key": "ACME" },
      "summary": "Add formatCurrency utility",
      "description": "Implement formatCurrency(amount, currency?) in src/utils.ts\n\nAcceptance Criteria:\n- Returns formatted string with currency symbol\n- Defaults to USD when currency is omitted\n- Handles negative amounts correctly"
    }
  }
}
```

Two fields drive the routing and the AI's work:

- `issue.fields.project.key` must exactly match the `jira_project_key` you registered. The gateway rejects the webhook if no project matches.
- `issue.fields.summary` and `issue.fields.description` become the AI's instructions. Write them the same way you'd write a ticket for a human engineer — clear acceptance criteria produce better results.

### Step 2 — Trigger the task

```bash
pnpm trigger-task --key ACME-1
```

The `--key` flag overrides `issue.key` in the payload. This matters because the gateway deduplicates on `external_id` — sending the same key twice skips the second task. Use a timestamp suffix to avoid collisions across runs:

```bash
pnpm trigger-task --key ACME-$(date +%s)
```

The script signs the payload with `JIRA_WEBHOOK_SECRET`, posts it to `POST /webhooks/jira`, then polls Supabase every 30 seconds until the task reaches `Done`. Sample output:

```
→ Sending webhook for key ACME-1609459200...
✓ Webhook accepted — task ID: a1b2c3d4-e5f6-7890-abcd-ef1234567890
→ Polling task status...
  [00:30] status: Executing
  [01:00] status: Executing  (stage: typescript)
  [02:30] status: Executing  (stage: unit)
  [04:00] status: Submitting
  [04:30] status: Done
✓ Task complete!
→ PR: https://github.com/acme/my-typescript-app/pull/42
```

### Step 3 — What the AI does (automatically)

Once the task is dispatched, the worker container handles everything:

1. Clones the registered repo (`git clone --depth=2`)
2. Runs the install command (`npm ci` in this example) inside the cloned workspace
3. Creates branch `ai/ACME-1609459200-add-formatcurrency-utility` from the default branch
4. Starts OpenCode (AI coding agent) on port 4096 inside the container
5. **Planning phase**: runs a dedicated OpenCode session that produces a structured plan file (`.sisyphus/plans/{TICKET_ID}.md`). The plan is structurally validated (must have at least one wave with at least one task).
6. **Plan judge gate** (if `PLAN_VERIFIER_MODEL` is set): sends the plan to the configured model for a rubric check (`scope_match`, `function_names`, `no_hallucination`). If rejected, rewrites the plan with corrective feedback and retries (up to 2 attempts total). If all attempts fail, the task transitions to `AwaitingInput`.
7. Executes the plan wave by wave — each wave is a separate OpenCode session implementing the tasks in that wave, followed by a git push
8. Runs the validation pipeline: TypeScript check, lint, unit tests, integration tests, e2e tests (only stages configured in the project registration run; unconfigured stages are skipped)
9. On any failure, re-prompts OpenCode with the error output and retries (up to 3 iterations per stage, 10 globally)
10. Commits the changes and opens a PR on GitHub

Typical duration is 10 to 30 minutes depending on ticket complexity, number of plan waves, and model response time. The planning phase adds roughly 2–5 minutes before coding begins.

### Step 4 — The PR

The PR lands on GitHub with:

- **Branch**: `ai/{ticketId}-{slug}` where `slug` is the ticket summary in kebab-case
- **Base branch**: the `default_branch` from the project registration
- **Title**: `[AI] {TICKET_ID}: {summary}` (e.g. `[AI] ACME-1609459200: Add formatCurrency utility`)

`GITHUB_TOKEN` must have push access to the repo. The token is passed into the worker container by the lifecycle function — it's the same token for all registered projects.

### Step 5 — Verify (optional)

```bash
pnpm verify:e2e --task-id <uuid-printed-by-trigger-task>
```

This runs a 12-point verification suite. The four most important checks:

1. Task status is `Done` in the database
2. A pull request exists on GitHub at the expected URL
3. Execution tokens were recorded (execution row exists with a `heartbeat_at`)
4. The worker container was cleaned up (no dangling container with that task ID)

## Managing Registered Projects

All five admin endpoints require the `X-Admin-Key: $ADMIN_API_KEY` header.

| Method | Path                | Description          |
| ------ | ------------------- | -------------------- |
| POST   | /admin/projects     | Register new project |
| GET    | /admin/projects     | List all projects    |
| GET    | /admin/projects/:id | Get one project      |
| PATCH  | /admin/projects/:id | Update fields        |
| DELETE | /admin/projects/:id | Remove project       |

```bash
# List all projects
curl http://localhost:3000/admin/projects \
  -H "X-Admin-Key: $ADMIN_API_KEY"

# Update the install command (useful when switching package managers)
curl -X PATCH http://localhost:3000/admin/projects/<id> \
  -H "Content-Type: application/json" \
  -H "X-Admin-Key: $ADMIN_API_KEY" \
  -d '{"tooling_config": {"install": "yarn install --frozen-lockfile"}}'

# Delete (returns 409 Conflict if tasks are in Ready, Executing, or Submitting state)
curl -X DELETE http://localhost:3000/admin/projects/<id> \
  -H "X-Admin-Key: $ADMIN_API_KEY"
```

PATCH accepts any subset of the project fields — only the fields you send are updated. You can update `tooling_config` without touching `repo_url` or `default_branch`.

## Troubleshooting

### `ADMIN_API_KEY not set`

**Symptom**: The wizard prints `✗ ADMIN_API_KEY not set — add it to .env or generate with: openssl rand -hex 32` and exits immediately.

**Root cause**: The `.env` file is missing `ADMIN_API_KEY`, or the file itself doesn't exist.

**Fix**:

```bash
pnpm setup    # auto-generates ADMIN_API_KEY and writes it to .env
# or manually:
echo "ADMIN_API_KEY=$(openssl rand -hex 32)" >> .env
```

---

### 409 Conflict on registration

**Symptom**: `pnpm register-project` or a curl POST returns HTTP 409. The wizard prints `✗ A project with Jira key ACME is already registered.`

**Root cause**: A project with that `jira_project_key` already exists. The key is unique per installation.

**Fix**: Find the existing project and either reuse it or delete it first (only if no active tasks):

```bash
# Find the existing project
curl http://localhost:3000/admin/projects \
  -H "X-Admin-Key: $ADMIN_API_KEY"

# Delete it (only if no tasks are in-progress)
curl -X DELETE http://localhost:3000/admin/projects/<id> \
  -H "X-Admin-Key: $ADMIN_API_KEY"
```

---

### `action: project_not_registered`

**Symptom**: `pnpm trigger-task` completes quickly with no task created. The gateway logs show `action: project_not_registered`.

**Root cause**: The `issue.fields.project.key` in `test-payloads/jira-realistic-task.json` doesn't match any registered project's `jira_project_key`. The comparison is case-sensitive.

**Fix**: Open `test-payloads/jira-realistic-task.json` and set `issue.fields.project.key` to exactly match the key you registered. Verify the registered key with `GET /admin/projects`.

---

### PR not created / task stuck in Executing

**Symptom**: The task reaches `Executing` but never moves to `Submitting`. No PR appears on GitHub. Container logs show a git push error or a GitHub API PR creation failure.

**Root cause**: `GITHUB_TOKEN` doesn't have push access to the registered repo.

**Fix**: Verify the token's permissions:

```bash
curl -H "Authorization: token $GITHUB_TOKEN" https://api.github.com/user
# or
gh auth status
```

The token needs `repo` scope (or `public_repo` for public repos) and must be a collaborator or org member with write access to the target repository.

---

### Wrong install command

**Symptom**: Worker container fails during the install step. Logs show `command not found` or a lockfile mismatch error (e.g., `ERR_PNPM_OUTDATED_LOCKFILE`).

**Root cause**: The registered `tooling_config.install` doesn't match the repo's package manager or lockfile.

**Fix**: PATCH the project with the correct install command:

```bash
curl -X PATCH http://localhost:3000/admin/projects/<id> \
  -H "Content-Type: application/json" \
  -H "X-Admin-Key: $ADMIN_API_KEY" \
  -d '{"tooling_config": {"install": "npm ci"}}'
```

If you also changed worker code to support the new package manager, rebuild the Docker image:

```bash
docker build -t ai-employee-worker:latest .
```
